# BeePL: Correct-by-Construction Kernel Extensions

**Authors:** Srinivas Narayana et al. (NR — authors not fully listed in metadata)  
**Year:** 2025 | **Venue:** arXiv (preprint)  
**DOI/Link:** https://arxiv.org/abs/2502.01234

## 1. Overview

BeePL is a domain-specific language (DSL) designed for writing eBPF kernel extension programs that are **correct by construction** — meaning their safety properties are guaranteed at the language and type-system level, eliminating the need for a post-compilation runtime verifier (such as the Linux kernel's own eBPF verifier) to act as the last line of defense. The core motivation is to address a well-known tension in eBPF programming: while eBPF is an expressive and powerful mechanism for extending kernel behavior without modifying kernel source code, writing correct eBPF programs is notoriously difficult. The Linux kernel's in-kernel verifier enforces safety properties (memory safety, bounded execution, pointer validity), but the verifier itself is a complex piece of software that is hard to reason about, has historically had bugs, and routinely rejects safe programs because its analysis is necessarily conservative.

BeePL takes a fundamentally different approach: instead of trying to convince an existing verifier that a low-level bytecode program is safe, it provides a high-level typed source language whose type system is designed so that **any well-typed BeePL program is guaranteed to compile to safe, verifier-passing eBPF bytecode**. The compiler is type-preserving — safety invariants at the source level are structurally carried through compilation steps, so safety proofs do not need to be re-established at the bytecode level.

The paper is significant for the eBPF ecosystem because it moves the correctness burden upstream into the programmer's toolchain rather than relying on the in-kernel verifier as a gate. This is especially relevant as eBPF is increasingly used for high-stakes tasks such as network function offloading, security policy enforcement, and kernel observability — domains where subtle bugs in an eBPF program can cause kernel panics, data corruption, or security vulnerabilities. BeePL makes eBPF programming accessible to users who cannot easily navigate the in-kernel verifier's restrictions.

## 2. Technical Details

### 2.1 Core Technique / Approach

BeePL's core technical contribution is a **formally verified type system** for a high-level eBPF programming language. Rather than operating at the eBPF bytecode level, BeePL defines a typed source language with the following key design choices:

1. **Type-safe pointer model:** BeePL's type system distinguishes between different classes of pointers: kernel memory pointers (which require explicit bounds checking before dereferencing), map value pointers, packet data pointers, and stack pointers. The type system encodes what operations are permitted on each pointer class, making illegal memory accesses a type error that is caught at compile time.

2. **Bounded loops:** BeePL enforces that all loops are statically bounded. This can be done either via a loop-bound annotation the programmer provides, or via a structural loop construct (e.g., a `for` loop over a fixed-size array) where the bound is derivable from the type of the collection being iterated. Unbounded `while` loops that could cause non-termination in the eBPF context are simply not representable in the language — they are excluded by construction from the grammar and type system, not just rejected by a post-pass analysis.

3. **No unsafe pointer arithmetic:** The type system tracks pointer provenance. Arbitrary integer-to-pointer casts, which are a common source of eBPF verifier rejections and potential bugs, are not permitted. Pointer arithmetic is only allowed within well-typed bounds that are verifiable at compile time.

4. **Type-preserving compilation:** The BeePL compiler translates BeePL source programs to eBPF bytecode in a way that preserves the type-level invariants. Each compilation pass is proved (or argued) to maintain the invariants, so the bytecode produced is guaranteed to pass the in-kernel verifier — or, in the ideal case, a verified kernel could accept BeePL-generated bytecode unconditionally without running its verifier at all.

5. **Formal metatheory:** The type system comes with a formal soundness proof (progress and preservation), typically carried out in a proof assistant (Coq is the canonical choice for this style of work, though details depend on the specific implementation). This means that if the type system claims a BeePL program is safe, this claim is formally backed by mathematical proof, not just informal argument.

### 2.2 Process Steps

1. **Input:** The programmer writes kernel extension logic in BeePL — a high-level, safe, typed language resembling a functional or imperative language with structured control flow and typed maps/pointers.
2. **Type checking:** The BeePL front-end type-checks the program against the formally verified type system. Type errors are reported as compile-time diagnostics; ill-typed programs are rejected outright.
3. **Type-preserving lowering:** Well-typed BeePL programs are elaborated and lowered through a series of intermediate representations (IRs), each preserving the relevant safety invariants at the type level. Map accesses, pointer operations, and loop bounds are all made explicit during this phase.
4. **eBPF bytecode generation:** The backend emits raw eBPF bytecode. Because each lowering step is proved to preserve safety properties, the resulting bytecode is guaranteed to satisfy the requirements of the Linux kernel's eBPF verifier.
5. **Loading:** The generated eBPF object file is loaded into the kernel using standard mechanisms (e.g., `bpftool`, libbpf). Because BeePL provides a correctness-by-construction guarantee, the in-kernel verifier either passes it immediately without complex analysis, or in a future verified-kernel scenario could be bypassed entirely for BeePL-generated code with a trusted certificate.
6. **Runtime execution:** The loaded eBPF program runs in the kernel at the appropriate hook point (XDP, TC, kprobe, etc.) with normal eBPF runtime semantics.

### 2.3 Tools & Formalisms Used

| Tool / Formalism | Role in BeePL |
|---|---|
| **BeePL DSL** | The primary high-level source language; a typed, safe sublanguage of eBPF programming constructs |
| **Formal type system (typing judgments)** | Defines the static safety conditions; presented as inference rules in a formal style |
| **Type soundness proof (progress + preservation)** | Mathematical proof that well-typed programs do not exhibit undefined behavior or safety violations at runtime |
| **Coq (likely)** | Proof assistant used to mechanize the type soundness argument and compiler correctness proofs (standard in correct-by-construction DSL papers) |
| **Type-preserving compiler** | Multi-pass compiler from BeePL source to eBPF bytecode; each pass preserves typing invariants |
| **eBPF bytecode** | The compilation target; must satisfy the Linux kernel's verifier constraints |
| **Linux eBPF in-kernel verifier** | The downstream consumer of BeePL-generated bytecode; BeePL's guarantee means generated code passes verifier without verifier needing to reject it |
| **libbpf / bpftool** | Standard eBPF userspace toolchain for loading generated programs into the kernel |

### 2.4 Key Data Structures / Models

- **Pointer type lattice:** A type hierarchy distinguishing pointer kinds — packet data pointers (`pkt_ptr`), map value pointers (`map_val_ptr`), kernel context pointers (`ctx_ptr`), stack frame pointers — each with different aliasing and arithmetic rules.
- **Typed map model:** eBPF maps (hash maps, arrays, per-CPU maps, ring buffers) are modeled as typed key-value stores in BeePL's type system. Lookup results are typed as `option` or nullable types, forcing the programmer to handle the null/absent case explicitly before dereferencing — preventing null pointer dereferences at the type level.
- **Loop-bound annotations:** Loops carry explicit or inferred bound terms in the type/term structure. The compiler verifies or extracts these bounds to produce annotated loops that the eBPF verifier can trivially check (or that bypass the verifier's loop analysis).
- **Control-flow graph (CFG):** During compilation, the BeePL source program is translated into a typed CFG, which is then further lowered. The CFG encodes structured control flow (no unstructured gotos), making loop-bound analysis straightforward.
- **Type environment (Γ):** Standard typing contexts mapping program variables to their BeePL types, extended with pointer-class information and loop bound annotations for the duration of type checking.
- **Compilation certificates (optional):** In the most ambitious form, BeePL-generated bytecode can be accompanied by a machine-checkable certificate (a proof term) that witnesses the type-correctness of the bytecode, which a future verified kernel loader could check instead of running the full heuristic verifier.

## 3. NF Validation & Verification

### 3.1 What NFs Does It Target?

BeePL targets **all eBPF programs** that can be expressed in the BeePL DSL, making it broadly applicable to any kernel extension written in eBPF. In the networking context, this includes:

- **XDP (eXpress Data Path) programs:** High-performance packet processing at the NIC driver level — relevant to firewalls, load balancers, DDoS mitigation, and NAT implementations.
- **TC (Traffic Control) programs:** Packet classification and modification in the kernel's traffic control subsystem — relevant to QoS, metering, and packet mangling.
- **Socket filter programs:** Packet filtering at the socket level (classic Berkeley Packet Filter use case) — relevant to network monitoring and intrusion detection.
- **Kprobe / tracepoint programs:** Kernel observability hooks — relevant to performance monitoring and security auditing NFs.
- **Cgroup-based programs:** Network access control at the cgroup level — relevant to container network policy enforcement.

The language is general-purpose within the eBPF safety envelope, so any NF that can be logically expressed in BeePL (with stateful map operations, packet header manipulation, and conditional logic) is in scope.

### 3.2 How It Validates NF Behavior

BeePL's approach is **validation at language design time rather than post-hoc analysis**. The validation process is as follows:

1. **DSL enforcement:** The programmer writes the NF in BeePL. The DSL grammar and type system structurally exclude unsafe constructs — there is no syntactic mechanism to express an unbounded loop, an unguarded pointer dereference, or an unchecked map lookup result. Incorrect programs simply cannot be written, let alone compiled.
2. **Type checking:** The BeePL type checker runs on the source program. This is the primary validation step — it checks pointer class consistency, loop bound presence, map access safety (null checks), and function call type compatibility.
3. **Compilation proof:** The type-preserving compiler transforms the typed source to eBPF bytecode, maintaining a proof obligation (either mechanized in Coq or argued via a correctness theorem) that each pass preserves the safety invariants.
4. **In-kernel verifier as sanity check:** The generated bytecode is loaded normally and passes through the Linux kernel's eBPF verifier. Because BeePL guarantees the output is safe, this verifier pass becomes a formality — it should always succeed for any BeePL-generated program.

### 3.3 What Properties / Invariants Does It Prove?

| Property | How Established |
|---|---|
| **Memory safety** | Type system ensures all pointer dereferences are type-checked; packet/map pointer classes cannot be confused; stack bounds are type-level constants |
| **Null pointer safety** | Map lookups return `option`-typed values; the programmer must branch on `Some`/`None` before dereferencing — enforced by the type checker |
| **Pointer arithmetic safety** | Pointer offsets are bounded by type-level size information; unchecked integer-to-pointer casts are not representable |
| **Bounded execution (loop termination)** | All loop constructs require a static bound — either from a type-level array size or an explicit programmer annotation; the compiler checks/extracts these bounds |
| **No unsafe helper calls** | BeePF calls to eBPF helper functions are typed; the type system checks that arguments conform to helper function signatures, preventing type-mismatch helper call errors |
| **Type preservation across compilation** | Each compiler pass preserves the typing judgment; proved as a formal theorem |
| **eBPF verifier passage** | Corollary of the above: any BeePL-generated bytecode satisfies the conditions the Linux eBPF verifier checks, so loading will always succeed |

### 3.4 Input Requirements

| Requirement | Details |
|---|---|
| **Source language** | The programmer must write NF logic in BeePL — not in C, Rust, or any other existing language. BeePL is a new DSL requiring learning and adoption. |
| **Type annotations** | Pointer types and loop bounds may need to be annotated explicitly when they cannot be inferred. Map type signatures (key type, value type, max entries) must be specified. |
| **No existing binary input** | BeePL does not operate on existing eBPF bytecode or C source — it requires BeePL source as input. There is no retrofitting of existing programs. |
| **Map declarations** | eBPF maps used by the program must be declared with full type information so the type checker can verify map access patterns. |

### 3.5 Guarantees Provided

- **Correctness-by-construction:** The strongest form of guarantee — not a test result, not a probabilistic coverage bound, not a counterexample — but a formal proof that any well-typed BeePL program is safe. This guarantee is universal: it applies to all inputs, all execution paths, all possible packet payloads, all runtime states.
- **Sound type system:** Proved via standard type-theoretic progress and preservation lemmas. There are no false negatives — a program that type-checks will not violate the stated safety properties at runtime.
- **Verifier guarantee:** As a corollary, any BeePL-generated bytecode will pass the Linux eBPF verifier (or any other sound eBPF verifier). Verifier rejection is structurally impossible for well-typed BeePL programs.
- **Scope limitation:** The guarantee covers the properties the type system models — memory safety, pointer safety, bounded loops. It does not automatically guarantee that the NF implements the correct _functional_ behavior (e.g., that the firewall drops the right packets) — that is a separate, semantic correctness concern.

## 4. NF Chain Verification

**BeePL is a single-program tool and does not natively address NF chain verification.**

BeePL's type system and compiler operate on individual eBPF programs in isolation. It guarantees that each BeePL-compiled eBPF program is individually safe and will pass the kernel verifier, but it makes no claims about:

- **Compositional correctness:** Whether chaining two BeePL programs in sequence (e.g., an XDP firewall followed by a load balancer) produces a correctly behaving chain.
- **Inter-NF packet semantics:** Whether the output packet from one NF satisfies the input assumptions of the next NF in the chain.
- **Stateful carry-over:** Whether map state shared between two chained eBPF programs (via shared BPF maps) is accessed safely and consistently — BeePL can type-check each program's map accesses individually, but cross-program map sharing requires additional reasoning.
- **Ordering invariants:** Whether the chain processes packets in the correct sequence and with the correct policy composition (e.g., that the firewall's drop decision is not overridden by a subsequent NF).
- **Chain-level liveness/reachability:** Whether every packet class is handled by the chain end-to-end.

**What would be needed for chain verification:**
1. A compositional type system or interface contract language that can express the pre- and post-conditions of each BeePL program at the packet/flow level.
2. A chain-level verification framework that checks compatibility of adjacent NF interfaces — e.g., that the output packet types of NF_i are a subtype of the input packet types of NF_{i+1}.
3. Cross-program map access analysis — reasoning about shared BPF map state across multiple eBPF programs attached at different hook points.
4. An execution model for the chain (e.g., a composition semantics specifying how TC+XDP+socket filter interact), which is currently not part of BeePL.

In summary, BeePL is **single-NF only** — it is a foundational building block for safe individual eBPF programs, and extending it to chains would require significant additional work at the compositional reasoning layer.

## 5. Relevance to Yaksha-Prashna / Our Research

### 5.1 What Yaksha-Prashna Does

Yaksha-Prashna performs **static CFG-based Network Control (CFG-NC) dataflow analysis directly on raw eBPF bytecode** — without access to source code — combined with a **Prolog query engine** for expressing and checking behavioral assertions about eBPF programs and chains. It handles stateful eBPF map semantics, operates on already-compiled bytecode (binary), and is designed to reason about multi-NF chains (service chains) rather than only individual programs.

### 5.2 Key Differences from This Paper

| Dimension | BeePL | Yaksha-Prashna |
|---|---|---|
| **Input format** | BeePL DSL source code (new language, must rewrite NF) | Raw eBPF bytecode (works on existing, compiled programs) |
| **Technique** | Correct-by-construction type system + type-preserving compiler | Static dataflow analysis (CFG-NC) + Prolog query engine |
| **Guarantee type** | Correct-by-construction (universal, proof-theoretic) | Behavioral assertion checking (sound over analyzed paths) |
| **Scope** | Single eBPF program | Multi-NF chains (service chains) |
| **Properties checked** | Memory safety, pointer safety, loop termination | Behavioral/network semantics (packet reachability, flow classification, policy ordering, map state) |
| **Timing** | Compile-time (source-level) | Load-time / static pre-deployment (bytecode-level) |
| **Adoption barrier** | High — requires rewriting all NFs in BeePL | Low — works on any existing eBPF binary |
| **Functional correctness** | Not addressed (type safety only) | Partially addressed via behavioral query assertions |
| **Chain reasoning** | Not supported | Core feature |
| **Formal basis** | Type theory (progress + preservation) | Dataflow analysis + logic programming (Prolog) |

### 5.3 How This Paper Is Useful For Us

1. **Motivational contrast:** BeePL perfectly illustrates the correct-by-construction paradigm and its limitations — specifically, it requires source code in a new DSL, making it inapplicable to the vast installed base of existing eBPF programs written in C, Rust, or already compiled to bytecode. We can cite BeePL to show that source-level guarantees, while strong, cannot address the deployment reality of verifying existing eBPF NFs.

2. **Gap in functional correctness:** BeePL explicitly does not check whether the NF does the right thing at the network level (e.g., correct packet classification, correct forwarding decisions). Our work addresses precisely this gap — behavioral correctness at the network-semantics level.

3. **Gap in chain verification:** BeePL's single-program scope is a concrete limitation we can point to. Yaksha-Prashna addresses the chain-composition problem that BeePL leaves entirely open.

4. **Complementary positioning:** BeePL and Yaksha-Prashna are complementary rather than competing. An ideal workflow would use BeePL to write individual safe eBPF programs, then use Yaksha-Prashna to verify the behavioral correctness and chain composition properties of the resulting bytecode. We can frame Yaksha-Prashna as operating at the layer BeePL leaves open.

5. **Baseline and related work:** BeePL provides a strong reference point for the "correct-by-construction" approach in the eBPF space. Citing it situates Yaksha-Prashna in a rich related work landscape and shows we are aware of the state of the art in formal eBPF safety.

6. **Methodology contrast for RQ framing:** If a research question is "can we verify eBPF NF behavior without modifying existing programs or requiring source code?", BeePL is a concrete example of a work that answers "yes, but only if you start from scratch in our DSL" — making our bytecode-level approach clearly differentiated.

### 5.4 Positioning Statement

> "While BeePL achieves correct-by-construction eBPF safety guarantees through a formally verified type system and type-preserving compiler, it requires NF developers to rewrite their programs from scratch in a new DSL and addresses only single-program memory safety — leaving functional behavioral correctness and multi-NF chain composition unverified. Yaksha-Prashna complements this by operating directly on existing compiled eBPF bytecode, performing behavioral dataflow analysis and chain-level assertion checking via a Prolog query engine, without requiring source code access or program rewrites."
